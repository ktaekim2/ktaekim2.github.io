## 1. Context / Problem

현재 회사 업무에서 Claude Code, Gemini CLI를 통해 업무 생산성을 대폭 향상시켰다. 단순히 프로그래밍 업무에만 활용할 것이 아니라 다방면으로 활용이 가능하다는 것을 깨달았다.

블로그는 언제나 내게 있어서 참 뭐랄까 막막한 것이었다.

프로그래머에게 블로깅이 도움이 된다는 것은 잘 알고 있다. 많은 개발 블로거, 프로그래밍 관련 서적에서 말하고 있고 나도 깊이 공감한다:
- 어려운 기술적인 부분을 내 생각으로 정리하는 데는 누군가를 가르치는 게 제일 좋다고 하듯이
- 머릿속으로 막연히 알고 있는 지식을 아웃풋을 만들어서 표현해야만 장기 기억이 되고 내 것이 된다
- 훌륭한 포트폴리오이자 마케팅 수단이기도 하다

그래서 프로그래밍을 처음 배운 학원에서도 만들었었고, 일을 하면서도 계속 시도했지만 **지속하기가 힘들었다**.

이유는 **시간!** 시간이 너무 많이 든다.

블로그는 기본적으로 퍼블릭에 공개되기 때문에:
- 정제된 글이어야 하고
- 정보 조사도 정확해야 한다 (거짓말을 할 수는 없으니까)
- 맞춤법, 띄어쓰기... 정말 쉽지 않다

그렇다보니 글 하나 작성하는 것에 대한 부담감은 엄청나다. "블로그 해야지" 하면서도 지금까지 이런 상황이다.

---

## 2. Constraints & Assumptions

- **목적**: 수익이 아니라 생각 정리 + 이직 시 포트폴리오
- **핵심 가치**: 내 생각과 기술적 관점을 보여주는 것
- **AI 활용 경험**: 이미 업무에서 Claude Code, Gemini CLI로 생산성 향상 경험
- **시간 제약**: 블로그에 과도한 시간을 쓸 수 없음
- **정제 필요**: 퍼블릭 공개이므로 최소한의 퀄리티는 필요
- **진정성**: 틀린 생각도 그대로 남아야 함 (내용 수정 불가)

---

## 3. Options Considered

### Option A: 기존 방식 (수동 블로깅)
- 모든 것을 직접 작성
- 장점: 완전한 통제, 글쓰기 능력 향상
- 단점: 시간 소모, 지속 불가능, 맞춤법/띄어쓰기 부담

### Option B: 블로그 포기
- 블로깅을 아예 하지 않음
- 장점: 시간 절약
- 단점: 사고 정리 기회 상실, 포트폴리오 없음

### Option C: AI 기반 블로깅 파이프라인
- 휘갈겨 쓴 메모 → AI 정제 → 커밋/푸시 자동화
- 장점: 저마찰 발행, 지속 가능, 영어 학습 병행
- 단점: AI 의존도, 글쓰기 능력 정체

---

## 4. Decision & Rationale

**Option C: AI 기반 블로깅 파이프라인**을 선택했다.

### 왜 이 방식인가?

AI를 활용해서 글을 정제하고, 자동으로 커밋과 푸시까지 가능하다면 블로깅에 대한 부담을 엄청나게 줄일 수 있다는 생각이 들었다.

나는 블로그를 하면서 수익을 노리는 게 아니다. 생각을 정리하고 나중에 이직하거나 할 때 내 생각과 기술적 관점을 사람들에게 보여줄 수 있다면 그걸로 된다고 생각하기 때문에.

그래서 프로그래밍에 활용하듯이 Claude Code를 붙이고, 프롬프트를 작성했다.

**핵심 원칙**:
- **AI가 내 글을 정제하되, 내 생각은 온전히 글에 남아야 한다** (틀린 생각도)
- 그래서 내용 자체의 수정은 못하도록 프롬프팅했다
- 내 기술적 생각을 영어로 표현할 수 있게 하기 위해 영어 버전과, 영어 학습에 도움될 만한 프롬프트도 추가했다

**구체적 효과**:
1. **시간 부담 해결**: 맞춤법, 띄어쓰기, 문법, 논리 재배치 → AI가 처리
2. **진정성 보존**: 내용은 내 머릿속에서 나옴, AI는 정제만
3. **영어 학습**: 한영 병행 + 전문 용어 매핑
4. **지속 가능성**: 부담이 줄어들면 지속할 수 있음

---

## 5. How: Implementation

### 5.1 CLAUDE.md 작성

Claude Code에게 작업 지침을 제공하는 `CLAUDE.md` 파일을 작성했다:

**핵심 원칙**:
- **내용 수정 금지**: 논리적 모순이 있어도 그대로 보존
- **형식 정제 필수**: 문법, 띄어쓰기, 논리 재배치, 톤 조정
- **한영 병행**: 동일 내용을 한국어, 영어로 작성
- **Meta 섹션**: 영어 표현 학습, 전문 용어 매핑, 구조 설명, 면접 피치 스크립트

### 5.2 Jekyll 설정

GitHub Pages에서 자동 빌드되도록 Jekyll 설정:
- `_posts/` 폴더에 마크다운 작성
- Front Matter로 메타데이터 관리
- `_config.yml`로 사이트 설정
- CSS로 미니멀한 디자인

### 5.3 워크플로우

```
1. 휘갈겨 쓴 메모 작성 (날것의 생각)
   ↓
2. Claude Code에게 전달
   ↓
3. AI가 CLAUDE.md 원칙에 따라 정제
   - 문법/띄어쓰기 수정
   - 한영 병행 작성
   - Meta 섹션 추가 (Jekyll 주석으로 숨김)
   ↓
4. Git 커밋 & 푸시
   ↓
5. GitHub Pages 자동 빌드
```

### 5.4 프롬프팅 전략

**금지 사항**:
- 내가 쓰지 않은 내용 추가
- 결론 발명
- 작가의 말투와 표현 방식 제거 (과도한 형식화)

**허용 및 권장 사항**:
- **명백한 사실 오류 수정**: "Python은 컴파일 언어다" → 고쳐줌
- **논리적 모순 지적**: 수정 여부는 내가 판단
- **문법, 띄어쓰기 교정**: 필수
- **논리 재배치**: 의미는 보존하되 읽기 쉽게
- **말투 보존**: "그렇다보니", "뭐랄까" 같은 표현 유지
- **작가의 색채 보존**: 사람마다 글쓰기에 체취가 남는다. 그걸 없애면 재미가 없다. 맞춤법은 정제하되 작가의 색채는 남긴다.
- **질문을 통한 사고 유도**: 생각할 거리 제공

### 5.5 영어 학습 전략

Meta 섹션의 영어 표현, 전문 용어 매핑은 학습 자료로 활용한다:
- 포스트 작성 후 Meta 섹션 복습
- AI와 음성 대화로 스피킹 연습 (별도 진행)
- 실효성 측정: 실제 영어 면접에서 검증

## 6. Trade-offs & Risks

### 포기한 것
- **글쓰기 능력 향상 기회**: AI에 의존하므로 직접 쓰는 능력은 정체될 수 있음
- **완전한 통제**: AI의 해석에 따라 의도와 다르게 정제될 수 있음

### 얻은 것
- **지속 가능성**: 부담이 줄어 실제로 블로그를 지속할 수 있음
- **영어 학습**: 내 생각을 영어로 표현하는 법을 배움
- **생산성**: 같은 시간에 더 많은 콘텐츠 생산 가능
- **사고 정리**: 날것의 생각을 구조화된 글로 변환

### 리스크
- **AI 오역**: 내 의도를 잘못 해석할 수 있음 (→ CLAUDE.md로 경계 설정)
- **진정성 의심**: AI가 쓴 글이라는 것을 밝혀야 하나? (→ 정제만 하므로 내 글로 봄)
- **기술 의존**: Claude Code가 없으면 못 쓰게 되는가? (→ 직접 쓸 수도 있음)

## 7. Outcome & Operational Principles

### 지금까지의 결과
- ✅ CLAUDE.md 작성 완료
- ✅ Jekyll 블로그 설정 완료
- ✅ ADR-0001, 첫 포스트 작성 완료
- ✅ Meta 섹션 숨김 처리 완료

### 운영 방침

**발행 주기**:
- 주말에 최소 1개 포스팅 목표 (평일은 업무로 힘들 것)
- 더 쓸 수 있으면 쓰되, 최소치는 주 1회
- 지속 가능성 지표: 포스팅 개수

**중단 시 판단**:
- 만약 이후 다시 중단하면, 이 파이프라인 구조에 뭔가 문제가 있었다고 판단
- 그때는 CLAUDE.md나 워크플로우를 재검토

**미완성 포스트 처리**:
- 커밋은 해놓되 발행은 안 함 (Jekyll의 `published: false` 옵션 활용)
- 나중에 완성되면 발행

**독자 반응 대응** (현재 댓글 기능 없음):
- 만약 댓글 기능을 추가하고 누가 지적하면 토론
- 내가 틀렸으면 수정하되, 처음 잘못된 포스팅은 남기고 밑에 주석 추가
- 수정 이력도 기록

**진정성에 대한 입장**:
- AI가 정제했지만 내 글이다
- 전체적인 흐름은 내 머릿속에서 나왔으니 내 글
- AI는 맞춤법, 문법, 구조를 정제할 뿐
- 면접관이 "AI가 쓴 거 아니냐"고 물으면:
  - **"맞다. 하지만 프롬프팅으로 원하는 결과를 설정하고, 그 결과를 검토·수정하는 과정은 지금 개발 업계에서 하는 바이브 코딩과 같다."**
  - 날것의 메모는 저장하되 공개 안 함 (정제된 것과 99% 일치하고, 솔직히 맞춤법 다 틀려서 창피하기도 하니까)

**CLAUDE.md 관리**:
- 고정 문서로 관리 (Claude뿐 아니라 Gemini도 사용 가능하게)
- 수정 시점: 작문 스타일 변화, 섹션 추가, 제약 사항 추가 등 필요할 때
- 파이프라인 자체 변화 기록은 필요시 고민

**블로깅 목적 우선순위**:
1. **사고 정리** (최우선)
2. 이직용 포트폴리오
3. 영어 학습
4. 장기 기억 형성

사고가 정리되면 그 내용을 바로바로 설명할 수 있다. 정리되지 않으면 아무리 AI가 예쁘게 글을 써줘도 내 생각이 아니다.

### 열린 질문
- AI 정제 후에도 내 목소리가 충분히 남아있는가?
- 영어 학습 효과가 실제로 있는가? (영어 면접에서 검증 예정)
- 이 방식으로 얼마나 오래 지속할 수 있을까?
- 면접관이 이 블로그를 보고 내 사고 방식을 이해할 수 있을까?

### 다음 단계
- 실제 기술 포스트 작성 (튜토리얼 아닌 사고 과정 기록)
- 3개월 후 회고: 지속 가능한가? 효과가 있는가?
- CLAUDE.md 지속적 개선 (필요시)

## Decision Summary (EN)

- Built an AI-driven blogging pipeline using Claude Code to reduce friction in technical writing
- Problem: Blogging is valuable but unsustainable due to time burden (grammar, fact-checking, formatting)
- Solution: AI refines raw notes while preserving authentic thinking (no content modification allowed)
- Implementation: CLAUDE.md for AI guidelines, Jekyll for static site, bilingual posts with learning-focused Meta sections
- Trade-off: Sacrificing writing skill improvement, but gaining sustainability and English learning opportunities
- Open question: Will this approach remain sustainable? Will interviewers understand my thinking through these posts?
